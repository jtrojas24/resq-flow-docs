## Shared path

`resq-flow` is a consumer of the shared local observability stack, not a parallel pipeline.

```text
resq-agent -> OTLP -> Vector -> VictoriaLogs
                           -> VictoriaTraces
                           -> VictoriaMetrics
                           -> resq-flow relay -> WebSocket -> UI
```

The core rule is simple:

- producers emit once
- `Vector` fans out
- Victoria still gets the full telemetry stream
- `resq-flow` only gets the filtered flow-shaped subset

This is fanout, not replacement.

## Responsibility split

- `resq-agent` emits the logs and traces that describe the real runtime boundaries
- `resq-fullstack` owns the applied local `Vector` config and the Victoria sinks
- `resq-flow` owns the relay, flow contracts, UI, and flow-aware presentation

That means the normal path is shared-stack first, not direct-to-`resq-flow`.

`Vector` only does generic routing. It should not carry the meaning of a specific flow.

## Inside resq-flow

`resq-flow` has two services:

- `relay/`
  a Rust service that receives the filtered logs and traces, applies the flow contract, and publishes normalized live updates
- `ui/`
  a Vite + React UI that renders graph-backed and headless flow experiences

This split matters because the relay owns the flow-aware telemetry contract and the UI owns presentation.

## Live and history

There are two intentional product paths:

### Live

`Vector -> relay ingest -> flow contract -> UI`

### History

`UI -> relay history API -> Victoria queries -> contract filter -> UI`

The important idea is consistency:

- live and history should use the same flow semantics
- the relay should stay the canonical flow filter
- the UI should not invent a different matching model

## Run-backed vs flow-visible activity

The relay keeps one normalized flow event model whether or not a record has `run_id`.

That means:

- producer-owned `run_id` marks one concrete execution story
- the Runs surface is built from run-backed events
- logs, canvas detail, and history can still show flow-visible events without `run_id`
- the UI should not synthesize fake run IDs just to force scheduler, polling, or mailbox-level activity into Runs
