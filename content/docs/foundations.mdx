---
title: Foundations
description: The rules that define what belongs in the graph, what stays as detail, and how drill-down should work.
---

The foundation docs exist to make graph semantics explicit before implementation starts.

The goal is to avoid two failure modes:

- everything important becomes a graph node
- `resq-flow` has to infer graph semantics from arbitrary telemetry richness

## The three buckets

### First-class nodes

These are real graph nodes. A thing is first-class when it is:

- an independently meaningful execution unit
- worth showing as its own node
- bound directly by producer code
- backed by a canonical span boundary
- worth lighting independently
- worth clicking for drill-down

Common first-class kinds:

- queue
- worker
- decision
- process
- store

### Node-attached details

These are important, but not first-class nodes.

They may appear as:

- grouped child rows
- detail-panel entries
- edge activity
- badges or substeps
- clickable subordinate items

They should not:

- own a canonical `component_id`
- require independent node identity
- light up as fake standalone nodes

### Non-graph telemetry

This telemetry is still valuable for search, debugging, logs, traces, and future views, but it should not shape graph presence.

## Hard rules

- Canonical producer binding decides first-class presence.
- Telemetry attached as stage, detail, or artifact information is not first-class.
- First-class node presence should come from canonical spans, not logs alone.
- Detail telemetry may animate edges, groups, and panels without becoming nodes.
- Real queue and worker boundaries should remain first-class by default.
- Process inspectors should be meaning-first, not just timing-first.
- Default node detail should favor meaningful and selected operational events over raw tracing exhaust.
- Errors, warnings, retries, stuck states, and manual intervention should surface even when they originate in low-level detail.

## Hierarchy model

`resq-flow` should treat the graph as a hierarchy:

### Level 1: first-class nodes

These are the independently meaningful units that support:

- canonical identity
- independent lighting
- status and timing
- click drill-down

### Level 2: grouped or node-attached details

These are subordinate items such as:

- enqueue actions
- internal substeps
- side-effect writes
- resource references
- exception-path notes

Default preference:

1. side panel
2. grouped child row or badge
3. annotation only when the graph is materially harder to understand without it

### Level 3: non-graph observability detail

This belongs in:

- detail panels
- logs
- trace views
- history and explorer views

## Drill-down model

Artifacts and resources are not automatically first-class nodes.

Usually they should be represented as:

- data attached to a first-class node
- related panels and details
- resource references
- edge or group-level context

Good examples:

- an S3 write shown under the process that performed it
- a Postgres mutation shown in the node inspector
- queue depth or enqueue metadata shown on a queue or queue-facing edge
- external call results shown on the calling worker or process

Keep the graph sparse. Keep the drill-down rich.

## Signal-first node detail

The default node experience should answer:

- what happened here
- why it mattered
- what changed downstream

That means the default canvas and log surfaces should agree on the same signal model:

- meaningful events by default
- selected operational events by default
- raw telemetry behind advanced or show-all views

Severity beats filtering. If an event represents real risk or a meaningful state change, it should not disappear just because it came from a low-level detail path.

## Node catalogs

A node catalog is the declared list of first-class graph nodes for a flow.

Each entry should be:

- stable
- directly bound by producers
- backed by a canonical span
- graph-worthy
- valuable to inspect

Naming guidance:

- first-class node IDs: short, semantic, kebab-case
- stage and detail IDs: dotted action names
- resource anchors: simple lowercase IDs like `postgres` and `s3`
- display labels: human-readable, not forced to match canonical IDs

## Bottom line

The graph should not be built by guesswork.

The better model is:

- producers emit intentional semantics
- flows declare first-class nodes intentionally
- details stay rich without becoming fake nodes
- `resq-flow` renders one projection of a broader observability system
