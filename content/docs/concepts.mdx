`resq-flow` is a flow-oriented observability product, not a graph toy, not a log viewer, and not just a metrics dashboard.

## Product identity

The intended navigation model is:

```text
Flows Home -> Flow (Canvas | Metrics | Logs) -> Run -> Node -> Logs/Spans
```

The graph is one view, not the whole product. Metrics mode, headless mode, and log-first mode are equal citizens.

## Flow model

Each flow has two layers:

- a shared JSON contract in `ui/src/flow-contracts/*.json`
- an optional TypeScript view config in `ui/src/flows/*.ts`

The JSON contract is the stable shared boundary. It defines:

- coarse telemetry matching rules
- context retention rules
- the flow identity and name used by the relay and UI

The TypeScript view config is optional. It adds rich view concerns such as:

- React Flow nodes and edges
- node and span mapping
- graph-specific presentation

If a flow has no TypeScript view config yet, it can still exist as a headless flow for history, logs, trace detail, and future non-graph views.

Today that means:

- the required JSON file is the shared flow contract
- the rich graph view, if present, still lives in TypeScript
- a new flow can ship as headless before it ships as a canvas

## Relay behavior

The Rust relay:

- receives OTLP traces and logs on port `4200`
- loads flow contracts from `ui/src/flow-contracts`
- matches incoming telemetry against those contracts
- keeps additional trace context according to each flow contract
- tags kept events with `matched_flow_ids`
- publishes WebSocket `snapshot` and `batch` envelopes for live clients
- applies the same contract-driven filtering model to history queries via `flow_id`

## Live and history

Two paths matter:

- `live` for watching execution happen now over WebSocket
- `history` for querying and comparing what already happened

`resq-flow` should support both without treating history as an afterthought.

## Flow semantics

When `resq-flow` renders a graph, it should bias toward operationally truthful execution boundaries and rich subordinate detail.

That means:

- real queue and worker boundaries stay first-class by default
- meaningful decisions and processes can also be first-class
- important writes, handoffs, validations, and state transitions usually stay visible as detail
- exception paths often belong in side branches or notes instead of the main happy path
- architecture sketches can show more than the final graph, but the product graph should stay honest about execution ownership

Two newer rules matter here too:

- process and detail should not collapse into the same inspector or visual treatment
- notes should default to the sidebar before they become canvas annotations

## Spec-first workflow

The preferred flow-author workflow is:

1. validate the real runtime or code path
2. classify each item as first-class, visible detail, or hidden detail
3. update the flow spec
4. implement producer and graph changes against that declared shape

For the concrete repo and file checklist, see [Adding a flow](/docs/adding-a-flow).
