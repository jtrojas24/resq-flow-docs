---
title: Logs, runs, and inspector
description: How traditional logs, run-level understanding, and node inspectors fit alongside the graph.
---

The graph is not enough by itself.

Users will naturally move between the visual flow, raw logs, and run-specific detail.

## Logs

Logs should be summary-first when possible.

The default row should answer "what happened?" before the user has to open raw telemetry.

For the current mail rollout, useful summaries include:

- `drafted; awaiting manual review`
- `drafted; awaiting manual approval`
- `auto-send approved; execution enqueued`
- `send queued`
- `sent`
- `retryable send failure`
- `terminal send failure`

Raw detail still matters.

Logs are still where you go for:

- exact messages
- structured attributes
- error context
- low-level debugging

But raw detail should not be the only readable layer.

The product should keep the source message, attributes, and tracing context available in drill-down.

## CLI and agent workflows

Logs also need a terminal-first path.

The CLI is the practical headless interface for:

- `status` checks before deeper debugging
- recent flow log inspection with `logs list`
- live log streaming with `logs tail`
- stable JSON or JSONL output for scripts and AI agents

This matters because many debugging loops start in a shell, not in a browser.
It also lets agents operate on a narrow, explicit surface instead of having to infer state from UI output.

The CLI should complement the UI, not replace it:

- use the CLI for fast operational checks and machine-friendly filtering
- use the UI when you need graph context, inspectors, and richer comparison across runs or nodes

## Runs and journeys

A useful flow product should let a user reason about one concrete execution path, not just one node at a time.

That means:

- following a run or journey through multiple nodes
- seeing what happened before and after an error
- understanding which boundary actually changed the outcome

## Inspector and sidebar

The inspector should be meaning-first.

For a process or node, it should prioritize:

- what this thing does
- recent outcomes and status
- related notes or caveats
- related resources and side effects
- useful recent logs

Timing is important, but it should not dominate the inspector before the meaning of the node is clear.

The first layer should explain the outcome.
The deeper layer can expose the raw telemetry.

## Live and history

These are two intentional paths, not one overloaded screen:

- `live`
  for watching execution happen now
- `history`
  for replaying or investigating what already happened

The product should use the same flow semantics for both.

## One filtering model

Canvas detail and default logs should share the same signal rules:

- summary-first and meaningful by default
- selected operational by default
- raw behind advanced or show-all

That keeps the product feeling coherent instead of teaching users two conflicting mental models.
