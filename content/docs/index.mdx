`resq-flow` is a local, flow-oriented observability product for understanding live execution and history across ResQ flows.

This site is the starting point for:

- local setup and day-to-day usage
- core product and flow concepts
- concrete flow-authoring guidance across `resq-agent` and `resq-flow`
- graph foundations and semantics
- Graph V2 principles and layout direction
- design-system guidance
- integration guidance for `resq-agent` and future consumers
- troubleshooting relay, ingest, replay, and docs-site issues

## What this docs site is for

- showing how to run and validate `resq-flow` locally
- documenting the product model behind flows, runs, nodes, and history
- making Graph V2, foundations, and design rules easier to navigate
- capturing how `resq-flow` integrates with the wider ResQ toolchain

## Core ideas

- `resq-flow` is not just a graph viewer
- the graph is one view of a broader flow-oriented observability system
- graphs should stay sparse while drill-down stays rich
- semantics should drive layout, not hand-placed coordinates
- producers should emit telemetry once into the shared observability path

## Current structure

- `Getting started` focuses on local setup, preferred runtime mode, and validation.
- `Concepts` explains the product model, flow model, and the role of live versus history.
- `Adding a flow` shows the concrete repo and file touchpoints for shipping a new flow.
- `Foundations` captures the rules that decide what belongs in the graph and how drill-down should work.
- `Graph V2` captures the layout and authoring direction for future graph work.
- `Design system` documents the UI language that supports the graph.
- `Integrations` explains how `resq-flow` fits into `resq-agent` and later consumers.
- `Troubleshooting` collects the practical debugging and verification paths.
