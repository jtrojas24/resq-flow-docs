`resq-flow` is most useful when it sits on the shared observability path rather than becoming a separate telemetry island.

## Current consumer: resq-agent

The current primary consumer is `resq-agent`.

The important integration shape is:

- producers send OTLP to Vector once
- Vector fans out to Victoria and `resq-flow`
- Victoria remains the storage and query source of truth
- `resq-flow` consumes the live stream and renders flow-oriented views on top

## Integration contract

Each flow uses:

- a shared JSON flow contract for matching and retention rules
- an optional TypeScript view config for rich graph presentation

This split matters because it lets a flow exist as:

- a rich graph-backed flow
- or a headless flow with history and logs first

Today the concrete shape is:

- `resq-agent` owns queue and worker truth plus emitted telemetry
- `resq-flow` owns the contract JSON, optional graph view, replay fixture, and flow-specific tests
- the only required JSON file on the `resq-flow` side is the shared contract
- the graph view, if present, still lives in `ui/src/flows/<flow-id>.ts`

## Vector fanout rules

The current preferred approach is intentionally coarse:

- do not fan out metrics to `resq-flow`
- fan out logs only when they include explicit flow-specific markers
- fan out traces when the batch contains stable markers that suggest the right flow family
- keep exact flow membership out of Vector

The relay owns exact contract matching and `matched_flow_ids`.

## Shared-stack expectations

When integrating another consumer with `resq-flow`, the default expectation should be:

- one shared OTLP path
- one storage source of truth
- one flow contract boundary
- optional richer graph presentation layered on later

## Adding a new flow

For a new flow, do not start by drawing boxes.

Start by:

1. writing the flow telemetry spec
2. making the runtime boundaries real in `resq-agent`
3. adding the shared flow contract in `resq-flow`
4. deciding whether the first release is headless or graph-backed
5. adding replay and tests

The concrete file-by-file checklist lives in [Adding a flow](/docs/adding-a-flow).

## Future consumers

The same model can support future `fullstack` usage without changing the core contract:

- producers still emit once
- flow identity still comes from contracts
- consumers can start headless and add graph views later

## Bottom line

`resq-flow` should plug into the wider ResQ toolchain as a projection on top of shared telemetry, not as a separate competing observability path.
